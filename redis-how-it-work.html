How Redis Works — Architecture Explained

1. In-Memory Storage

Redis stores all data in RAM instead of disk. This eliminates disk seek latency and enables extremely fast reads and writes.

Latency: sub-millisecond (<1ms) Storage: RAM Performance: extremely fast

2. Single-Threaded Execution

Redis processes all commands using a single thread. This avoids locks, race conditions, and context switching overhead.

SET key value GET key INCR counter (All executed sequentially)

3. I/O Multiplexing

Redis handles thousands of connections using epoll (Linux) or kqueue (macOS). One thread efficiently manages many clients asynchronously.

Clients → epoll → Redis Thread → Response

4. Optimized Data Structures

Redis uses specialized data structures for optimal performance depending on use case.

String → Simple values Hash → Objects List → Queues Set → Unique values Sorted Set → Ranking Bitmap / HyperLogLog → Analytics

5. Pipelining

Multiple commands can be sent in one network round-trip. This significantly improves performance.

Without pipeline: Client → Redis → Client → Redis With pipeline: Client → Redis (batch commands)

6. Persistence

Redis supports durability via disk persistence.

RDB → Snapshot periodically AOF → Append every write Hybrid → Recommended production mode

7. Replication

Redis uses master-replica architecture. Master handles writes. Replicas handle reads and redundancy.

Master → Replica 1 → Replica 2 → Replica N

8. Clustering

Redis Cluster distributes data across multiple nodes using hash slots. Total slots: 16384

hash(key) % 16384 → slot → node Example: user:1 → slot 100 → Node A user:2 → slot 8000 → Node B

9. High Availability (Redis Sentinel)

Redis Sentinel monitors nodes and performs automatic failover.

Master down → Sentinel detects Sentinel promotes Replica → New Master

10. Typical Production Architecture

Application │ Redis Cluster ├── Master 1 → Replica ├── Master 2 → Replica └── Master 3 → Replica Sentinel monitors cluster
